# Hytta

A sleek, modern cottage-booking app — three colour-themed bedrooms, optional
bed-level reservations in the red room, full-cottage bookings, calendar-driven
availability, and authentication via email/password, Google, and Apple (all
OIDC).

The app itself is **Hytta**; each deployment names its own cottage on first run
(see [First-run setup](#first-run-setup)). That name — e.g. "Granli" — is
what users see in the header, the browser title, calendar feeds and emails.

## Stack

- **Next.js 16** (App Router, React 19.2, Turbopack default).
  Uses `proxy.ts` (Next 16 replacement for `middleware.ts`).
- **TypeScript** (strict, `noUncheckedIndexedAccess`).
- **Tailwind CSS 4.1** — CSS-first configuration via `@theme` in
  `src/app/globals.css`. OKLCH colour tokens, no `tailwind.config.ts`.
  `tw-animate-css` for keyframes.
- **shadcn/ui-style primitives** built on Tailwind v4 (Button, Card, Input,
  Label, Badge) — local files, no runtime registry.
- **Drizzle ORM** (`drizzle-orm/bun-sqlite`) on a local **SQLite** file
  (`./data/hytta.db`, WAL + foreign keys + busy-timeout).
- **Auth.js v5** (`next-auth@beta`) with the Drizzle SQLite adapter, JWT
  session strategy, and Google + Apple + Credentials providers.
- **next-intl v4** for i18n (Norwegian Bokmål default, Oxford English).
- **Bun** as the runtime + package manager + test runner.
  All Next.js commands are run via `bun --bun next ...` so server code can
  use `bun:sqlite` natively.
- **argon2id** for password hashing (OWASP parameters).
- **react-day-picker v9** + `date-fns` for the calendar.
- **framer-motion** for tasteful transitions.

## Internationalisation

Two locales:

| Code    | Name              | Notes                                                |
| ------- | ----------------- | ---------------------------------------------------- |
| `nb-NO` | Norwegian Bokmål  | **Default.** First locale you land on.               |
| `en-GB` | English (Oxford)  | British vocabulary (`colour`, `centre`) + `-ize`.    |

The `en-GB` content follows **Oxford spelling** — see
`src/i18n/messages/OXFORD_SPELLING.md`. A test in `tests/i18n/parity.test.ts`
fails CI if any `-ise` verb (`organise`, `realise`, …) or AmE word
(`color`, `center`, …) leaks in, and verifies key parity with `nb-NO`.

Locale routing is done by next-intl through `src/proxy.ts` (the Next.js 16
proxy file — formerly `middleware.ts`). Navigation uses the typed
`Link` / `redirect` / `useRouter` from `src/i18n/navigation.ts`.

```

├─ i18n/                          # routing.ts, request.ts, navigation.ts, messages/
├─ app/
│  ├─ layout.tsx                  # passthrough root
│  └─ [locale]/
│     ├─ layout.tsx               # NextIntlClientProvider + ThemeProvider + chrome
│     ├─ page.tsx                 # landing
│     ├─ (auth)/login,register/
│     ├─ book/                    # the booking experience
│     └─ reservations/            # "my bookings"
├─ components/
│  ├─ ui/                         # Button, Card, Input, Label, Badge
│  ├─ booking/                    # DateRangePicker, CottageMap, ReservationSummary, AvailabilityLegend
│  ├─ auth/SignInButtons.tsx
│  ├─ Header.tsx, Footer.tsx, LangSwitcher.tsx, ThemeToggle.tsx, theme-provider.tsx
├─ db/
│  ├─ schema.ts                   # Drizzle SQLite tables + CHECKs + indexes
│  ├─ client.ts, drizzle.ts       # D1 client + drizzle wiring
│  ├─ demo.ts                     # optional fictional demo-data loader
│  └─ migrations/0000_init.sql    # generated by drizzle-kit
├─ lib/
│  ├─ auth/{config,handlers,password,apple-secret}.ts
│  ├─ booking/                    # types, validators, conflicts, availability, reservation-service, errors
│  ├─ env.ts                      # zod-parsed boot-time env validation
│  └─ utils.ts
└─ server/actions/                # 'use server' wrappers (reservations, availability, auth)
tests/
├─ booking/{conflicts,availability}.test.ts
├─ i18n/parity.test.ts
└─ helpers/db.ts                  # in-memory SQLite test harness
```

**SOLID alignment:**
- `ConflictChecker` and `ReservationService` are explicit, testable
  abstractions; the service receives its DB via constructor injection so
  tests inject an in-memory DB.
- The polymorphic `reservation` table is closed for modification but open
  for new `target_kind` values.
- Server actions own one responsibility each (auth, availability,
  reservations) and depend on the service rather than touching Drizzle
  directly.

## Booking conflict logic

A single `reservation` table with a `target_kind` discriminator
(`FULL_COTTAGE | ROOM | BED`) and a half-open `[check_in, check_out)` range
stored as ISO `YYYY-MM-DD` text. Six conflict cases are handled in one SQL
predicate (`src/lib/booking/conflicts.ts`):

| Existing → / Requested ↓ | FULL_COTTAGE | ROOM   | BED                    |
| ------------------------ | ------------ | ------ | ---------------------- |
| FULL_COTTAGE             | yes          | yes    | yes                    |
| ROOM                     | yes          | same id| bed inside that room   |
| BED                      | yes          | bed's parent room | same id     |

Inserts run inside a SQLite **`BEGIN IMMEDIATE`** transaction. With WAL +
`busy_timeout`, this serialises writers — concurrent attempts to book the
same range are guaranteed to produce exactly one success and one
`ConflictError`, without any application-level lock. The schema's `CHECK`
constraints (`reservation_target_shape`, `reservation_iso_dates`,
`reservation_dates_ordered`) are the last line of defence.

22 `bun:test` tests verify every conflict case, the half-open back-to-back
behaviour, cancellation, and validation rules.

## Security

- HTTP-only secure cookies via Auth.js v5; CSRF built-in; JWT session.
- argon2id (memoryCost 19456, timeCost 2, parallelism 1).
- Boot-time `zod` env validation (`src/lib/env.ts`); the app refuses to
  start unless every required variable is set.
- All server actions re-check the session and re-validate input with zod.
- Strong response headers (`next.config.ts`):
  CSP (with explicit allowances for Google + Apple OIDC),
  `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`,
  `Referrer-Policy: strict-origin-when-cross-origin`,
  `Permissions-Policy` denying camera/mic/geolocation/topics.
- Database CHECK constraints + `BEGIN IMMEDIATE` make double-bookings
  impossible regardless of application bugs.
- `data/`, `*.db`, `.env*.local` are gitignored. `.env.example`
  documents every required variable; secrets are never logged.

## Quick start

Prerequisites: **Bun ≥ 1.2** (only Bun is required — no Node, no Docker).

```bash
bun install
cp .env.example .env.local          # then fill in real values
bun run db:migrate                  # apply schema to the local D1 (starts EMPTY)
bun run demo                        # OPTIONAL: load fictional demo data to explore
bun run dev                         # http://localhost:3000
```

A fresh database is **empty** — exactly what a real deployment starts with. On
first run the app sends you to `/setup` to name the cottage and become the first
admin, then to create rooms. `bun run demo` is only for local exploration: it
loads an invented cottage ("Granheim"), a handful of `@example.com` members,
rooms, bookings, and chores so you can click around without setting everything
up by hand. It is never run on a deployment.

Cloudflare deployment is split into a real build phase and a deploy phase. The
deploy phase uploads the already-built Worker output; it does not rebuild.

### First-run setup

The cottage has no name baked into the code. The first time you open a freshly
migrated instance you are sent to **`/setup`**, where you give the cottage its
display name (e.g. `Granli`). It is stored as a singleton row in
`cottage_settings` and from then on drives the header brand, the browser title,
calendar feeds, and the From-name and body of every outbound email.

`/setup` is reachable without signing in (a brand-new instance has no users
yet) but refuses to overwrite a name once one exists, so it's a true one-shot.
Both the public auth pages and the signed-in area redirect to `/setup` until a
name has been set.

### Demo credentials

After `bun run demo`, sign in at `/login/credentials` with any demo member
(all share one password):

```
email:    astrid@example.com   (admin + manager)
          henrik@example.com   (member)
          maja@example.com     (member)
          jonas@example.com    (member)
password: demohytta2026
```

## Required environment variables

Copy `.env.example` to `.env.local` and fill every value. The app refuses
to boot otherwise.

| Var                   | Source / notes                                                |
| --------------------- | ------------------------------------------------------------- |
| `HYTTA_WORKER_NAME`   | Cloudflare Worker project name, e.g. `bekkeholt`             |
| `HYTTA_D1_DATABASE_ID` | Cloudflare D1 UUID from `wrangler d1 create hytta`           |
| `AUTH_SECRET`         | `openssl rand -base64 32`                                     |
| `AUTH_URL`            | Public origin in production, e.g. `https://hytta.example.com` |
| `EMAIL_PROVIDER`      | `resend` for Cloudflare Workers                              |
| `EMAIL_FROM`          | Verified sender, e.g. `Hytta <noreply@example.no>`           |
| `RESEND_API_KEY`      | Resend API key for real email delivery                       |
| `ADMIN_EMAILS`        | Optional comma-separated admin auto-promotion list            |
| `TEST_USER_*`         | Optional env-managed test account                            |

`HYTTA_WORKER_NAME` and `HYTTA_D1_DATABASE_ID` are not committed to Wrangler
config. Remote scripts read them from `.env.local` (or the shell environment),
generate ignored `wrangler.local.jsonc` files, and pass those files to
Wrangler/OpenNext.

### Cloudflare Workers deployment

Use **Cloudflare Workers Builds**, not a static Pages build. Configure the
GitHub deployment with these commands:

| Field          | Command          |
| -------------- | ---------------- |
| Build command  | `bun run build`  |
| Deploy command | `bun run deploy` |

`bun run build` generates the account-local Wrangler config from
`HYTTA_WORKER_NAME` and `HYTTA_D1_DATABASE_ID`, then runs
`opennextjs-cloudflare build`, producing `.open-next`.

`bun run deploy` deploys the Booking Durable Object worker, then runs
`opennextjs-cloudflare deploy` for the already-built app worker. If `.open-next`
is missing, deploy fails instead of building. Cloudflare Workers do not have a
long-running runtime start command after deployment.

For a project named `bekkeholt`, set `HYTTA_WORKER_NAME=bekkeholt`. The app
Worker deploys as `bekkeholt`, and the Durable Object worker deploys as
`bekkeholt-booking-do`. The cottage display name is still set later in `/setup`
and stored in D1.

## Verification checklist

1. `bun install`, `bun run db:migrate`, `bun run demo`.
2. `bun test` — booking + i18n test suites pass.
3. `bun run build` — production build is clean.
4. `bun run dev`, then in a browser:
   - `/` redirects to `/nb-NO` (default locale).
   - Sign in as `astrid@example.com` / `demohytta2026`.
   - The demo loads upcoming bookings spanning a room, a single bed, a slot, a
     whole-cottage week, and a guest stay — pick those dates to see rooms/beds
     and the "whole cottage" toggle disable as expected.
   - One dugnad chore shows as completed by a member other than its creator.
   - Try to book BLUE on `2026-05-16` — see a friendly conflict message.
   - Cancel a reservation on `/reservations` — its dates open back up.
   - Switch language to **English (UK)** — page rerenders in Oxford English,
     calendar weeks still start Monday, plurals localise (`1 night` /
     `2 nights`).
5. Concurrency check (optional): two parallel `curl` requests for the same
   target/range — exactly one succeeds, the other gets a `CONFLICT` error.
